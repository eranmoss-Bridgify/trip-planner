Bridgify Integration · Internal Report · May 2026

API Gap Report

Gaps and limitations encountered while building a trip-planning product on top of Bridgify. Written as feedback to the Bridgify team.

1 — Search Quality Gaps
A-01 Search result ordering is silently ignored High

Every sort parameter we tested — rating, -rating, price, best_seller, relevance, title — returns results in the same order regardless of the value. The API accepts these parameters without error but has no effect. There is no way to sort results server-side.

Impact: Results appear in supplier ingestion order. High-rated or cheapest products are not surfaced first. Any meaningful sort must be done client-side on whatever page of data was returned.

✓ Verified against production API — identical result order with and without ordering=-rating

A-02 Accented characters silently return zero results High

text_search is accent-sensitive. Accented input returns no results; removing accents returns full results.

QueryResults
"Sagrada Família" (with accent)0 results
"Sagrada Familia" (no accent)319 results
"Güell"0 results
"Guell"151 results
"batlló"0 results
"batllo"50 results

Impact: Any user who copies a destination from Google Maps, Wikipedia, or a search engine gets a silent empty grid with no error message.

Note: Other Bridgify consumers handle this by stripping accents client-side before sending the query — so this appears to be a known limitation rather than an unacknowledged bug. We apply the same workaround. Ideally this normalisation would happen server-side so every consumer doesn't have to solve it independently.

✓ Verified against production API

A-03 Text search is title/description match only — no category or intent awareness High

text_search performs a naive substring match against product titles and descriptions. There is no synonym resolution, no intent mapping, and no category awareness. Querying a broad user intent returns sparse or zero results because the word does not appear in product titles.

User IntentObvious KeywordResults (Barcelona)

Impact: We maintain a hardcoded keyword map (VIBE_TO_SEARCH) that translates user-facing categories into API terms that happen to return results in Barcelona. This map is city-specific and breaks for other destinations. Wellness, nightlife, and outdoor activities are effectively invisible unless the supplier used those exact words in the product title.

Also: Products have a category field (e.g. CULTURE, OUTDOOR), but the search endpoint does not accept category as a filter parameter. There is no way to search by category even though the data is there.

✓ Verified against production API — category_id filter tested with Museums (id=9), returned same count as unfiltered

A-04 Geo / bounding box filtering is accepted but silently ignored Medium

The search endpoint accepts geo parameters (lat_min, lat_max, lng_min, lng_max, bbox) and returns HTTP 200 — but the result count is identical to an unfiltered city-level search. These parameters appear to be dead code.

Impact: Cannot build a map view that only shows activities within the visible viewport. There is no proximity search, no radius filter, and no nearby endpoint. Client must fetch the entire city catalog and cluster coordinates in the browser.

✓ Verified against production API — tight Barcelona bbox returned count: 2856, same as unfiltered

2 — Duplicate Products
B-01 page/page_size params silently return wrong results — offset/limit is the correct approach Documentation Gap

This turned out not to be a Bridgify bug. Pagination works correctly using offset/limit — each page returns a unique set of results. The page/page_size params (a very common REST convention) are silently broken — every page returns the same results with no error. The next URL in the response already uses offset/limit, which is the hint we missed.

Recommendation: Either make page/page_size work, return a 400 when they're passed, or document clearly that offset/limit is the only supported pagination style.

B-02 Same tour listed by multiple suppliers with no grouping High

The same real-world attraction appears as separate products per supplier with no indication they are the same experience. A user searching "Sagrada Família" sees 324 listings from GetYourGuide, Viator, and Tiqets at prices ranging from $3.59 to $108.85 — a 30× price range for the same monument, with no grouping or "show cheapest" signal.

Supplier distribution in a 100-product sample for Barcelona "tour": Viator 54%, GetYourGuide 36%, Tiqets 8%, HotelBeds 2%. Viator products dominate results regardless of quality.

3 — Availability API Gaps
C-01 Availability returns slots but products are not orderable in sandbox Sandbox?

GET /attractions/products/availability/{id}/ returns date and time slot data for most products. The issue is that products with available slots cannot be booked — the booking step fails even when the calendar shows dates. The sandbox does not clearly distinguish between "available to view" and "available to order," which makes it impossible to build a reliable end-to-end booking flow in sandbox.

Impact: We can display availability calendars and slot pickers, but cannot confirm whether a product is truly purchasable without attempting a booking and receiving a failure. This may be a sandbox-only limitation.

4 — Endpoints That Exist But Are Incomplete or Missing
D-01 Best Sellers endpoints ignore city filter — return global or wrong-city results High

GET /attractions/best-sellers-bridgify/ and GET /attractions/best-sellers-gyg/ both exist but cannot be used as a city-specific editorial feature. Tested against Barcelona:

best-sellers-bridgify (no city)          → 504 timeout
best-sellers-bridgify ?city_name=Barcelona → 200, returns Disneyland Paris & London products
best-sellers-gyg ?city_name=Barcelona     → 200, returns Singapore & global products
best-sellers-gyg ?city=Barcelona         → 200, identical results (city param ignored)

Both endpoints return a static global list regardless of city. We fall back to a text_search=tours&city_name={city} query for the home page featured section instead. ✓ Verified against production API.

D-02 City / Destination Autocomplete Medium

GET /attractions/cities/ returns a list with lat/lng and attraction counts, but there is no query parameter for prefix matching or partial search. Building a usable autocomplete requires fetching the full city list and filtering client-side, which is impractical for a large list.

D-03 Category filter broken — 33 categories exist but only 1 works on search High

Bridgify exposes a well-structured taxonomy of 33 categories via GET /attractions/categories/ — Nightlife, Wellness & Wellbeing, Outdoor Activities, Museums, Shows & Performances, Family Friendly, Hidden Gems, and more. Products are tagged with these categories. There is even a category_main filter parameter on the search endpoint that accepts category slugs.

We tested all 33 category slugs against GET /attractions/products/?city_name=London&category_main={slug}. Only one returns a filtered result:

nightlife              → 85 results  ✓
wellness-&-wellbeing    → 0 results   ✗
outdoor-activities     → 0 results   ✗
museums                → 0 results   ✗
shows-&-performances   → 0 results   ✗
family-friendly        → 0 results   ✗
… 26 more categories all return 0

The infrastructure is there. The taxonomy is well-designed. But the filter only works for one category, which appears accidental. This makes the entire category system unusable for search filtering. ✓ Verified against production API.

D-04 No POI registry or landmark grouping Medium

Every product has a geolocation lat/lng — map pins are technically possible. But there is no landmark registry, no proximity endpoint, and no way to ask "show me all products at the Sagrada Família." Without a POI layer, a map would show 324 separate pins for the same monument.

Endpoints returning 404: /attractions/pois/, /attractions/landmarks/, /attractions/geo/, /attractions/products/nearby/, /attractions/products/clusters/
D-05 Search autocomplete / suggestions Low

GET /attractions/search-suggestion/?q=... exists but was not evaluated in detail. A type-ahead endpoint that returns matching product titles, category names, and city names would substantially improve search UX.

D-06 No global catalog export — impossible to build a local index without city-by-city crawling High

There is no endpoint to fetch all products across all cities, no catalog dump, no bulk export, and no webhook/feed for new or updated products. The only way to build a local copy of the catalog — required for deduplication, semantic search, or any offline processing — is to:

  1. Fetch the full city list from GET /attractions/cities/ (~1,000+ cities)
  2. Paginate through every city's products with GET /attractions/products/?city_name={city}&offset={n}
  3. Stitch, deduplicate, and store locally

This approach requires thousands of API calls, has no delta/incremental update mechanism, and provides no signal when a product is added, updated, or removed. Any local index goes stale immediately with no way to know.

Impact: Duplicate detection, semantic search, category mapping, and AI pre-filtering all require a local product index. Without a catalog export or change feed, building and maintaining one is impractical at scale.
5 — AI Assistant — API Gaps That Limit Quality

The AI assistant (Claude) proposes activities based on the user's trip context. Its quality is directly limited by the Bridgify search gaps documented above. This section connects those gaps to their AI impact.

How it works today: User sends a message ("I want something romantic") → server classifies intent → runs 2–3 parallel Bridgify keyword searches → injects top 15 results into Claude's prompt → Claude picks from that pool. Claude can only propose what the keyword search surfaces.
F-01 No date-filtered search — AI proposes activities regardless of availability on the trip dates High

The Bridgify search endpoint has no date parameter. There is no way to ask "show me activities available on June 15th" — the search always returns the full catalog regardless of whether products are bookable on the user's travel dates. Claude ends up proposing activities that may be fully booked, closed, or seasonal.

The per-product workaround is too slow: checking availability requires one API call per product (GET /availability/{id}/). Verifying 15 candidates before responding would add 15 HTTP calls to every AI reply — 3–8 seconds of extra latency minimum. In sandbox, even products with available slots often cannot be booked (see C-01).

What we need from Bridgify: a date filter on the search endpoint (?available_on=2026-06-15) or a batch availability check (POST /availability/batch with multiple IDs + a date range) so all candidates can be filtered in one round-trip before Claude sees them.

F-02 Keyword search limits proposal quality — AI inherits all search gaps High

Claude's proposals are only as good as what the keyword pre-fetch returns. When a user asks for "wellness" or "outdoor activities", Bridgify returns 0–2 results (see A-03), giving Claude almost nothing to pick from. The AI falls back to generic tour results — which is why proposals often feel irrelevant to the actual request.

The right solution is simple: map user intent to one of Bridgify's 33 categories — "I want something relaxing" → Wellness & Wellbeing, "outdoor" → Outdoor Activities — and pass the slug to category_main on the search endpoint. The taxonomy is well-designed and the parameter exists. But 32 of 33 categories return 0 results (see D-03). Until the filter works, intent-to-category mapping has nowhere to land.

F-03 No geo-aware proposals Medium

A user asking "find something near the Sagrada Família" or "activities in the Gothic Quarter" gets city-wide results — Bridgify's geo filter params are silently ignored (see A-04). Claude cannot honour location-specific requests.

AI AI Chat Route

POST /api/ai-chat — Claude (claude-sonnet-4-6) with server-side Bridgify pre-fetch.

Flow: User message → regex intent classification → parallel Bridgify keyword searches (2–3 terms) → inject top 15 results into system prompt → Claude proposes matching activities as confirmable cards. Claude does not call Bridgify directly — all search happens before the LLM call.

Tools: propose_add_activity, propose_remove_activity. Chat history persisted to localStorage (chat_ltm_{userId}_{tripId}, 60-message cap).

Gap: Intent classification is regex-based (23 rules). Compound requests, non-English input, and vague queries fall through to a default keyword set. This is a stopgap until semantic search is available.

7 — Recommendations

Expected behaviours and missing capabilities — what we'd expect from a mature travel activities API.

R-01 Semantic search Missing

Searching "romantic outdoor adventure" or "nightlife" should return relevant results even when those words don't appear in a product title. Expected: intent-aware search with synonym resolution and category mapping. Current behaviour is a raw text match — "wellness" returns 0 results, "nightlife" returns 1.

R-02 POI / Venue-level API Missing

A landmark registry where products are grouped under a canonical venue — "Sagrada Família", "Park Güell", "Louvre". Expected: GET /attractions/pois/ returning venues with linked products, and the ability to query GET /attractions/products/?poi_id=.... Currently there is no venue concept — the same monument appears as 324 separate products with no grouping.

R-03 Category filtering on search Missing

Expected: GET /attractions/products/?category_id=12 returns only products in that category. The 33 categories exist and products are tagged — but category is not an accepted filter on the search endpoint. Every consumer has to maintain their own keyword workarounds to approximate this.

R-04 Functional result ordering Missing

Expected: ?ordering=rating, ?ordering=price, ?ordering=-price return results sorted accordingly. All ordering parameters are currently accepted but silently ignored — results always return in supplier ingestion order.

R-05 Deduplication — one canonical result per attraction Missing

Expected: a main_product flag or canonical_group_id on each product so consumers can show one result per real-world attraction and let the user choose which supplier to book through. Currently 324 variants of Sagrada Família appear as separate results with no grouping signal.

R-06 Geo / proximity search Missing

Expected: GET /attractions/products/?lat=41.4&lng=2.17&radius_km=1 returning activities near a location. Every product already has geolocation. The bounding box params (lat_min, lat_max, etc.) are accepted but have no effect.

R-07 Special character / accented text search Missing

Expected: "Güell", "Sagrada Família", and "batlló" return the same results as their plain-text equivalents. Currently any accented or special character input returns zero results with no error — a silent failure.

R-08 Consistent availability response for open-voucher (BSN) products Missing

Expected: BSN products return { type: "BSN", slots: [] } from the availability endpoint — not a 400 or 404. availability_type is already in the search response; the availability endpoint should honour it.

R-09 City autocomplete Missing

Expected: GET /attractions/cities/?q=bar returns prefix-matched cities. The current endpoint returns the full unfiltered list — not usable for a type-ahead input.

R-10 Pagination correctness and stable ordering Bug

Expected: page 1 and page 2 return different products. Currently 50% of results are duplicated across pages. Any "load more" feature immediately shows repeated products.

R-11 Booking flow — documented payload + sandbox stability Bug

Expected: the customer-info step documents its accepted field names, and creating an order in sandbox does not fail with an ORM ambiguity error. These two issues block end-to-end booking testing entirely.

8 — Booking Flow — Cart API
Note: Some of the issues below may be specific to the sandbox environment and may not reflect production behaviour. We were unable to verify because the sandbox blocks end-to-end booking testing. Items marked Sandbox? are suspected sandbox-only; items marked Confirmed were reproducible across multiple sessions.
E-01 Cart UUID in request body is silently ignored — creates a new cart instead Confirmed

Adding an item to a cart requires the cart UUID in the URL path (POST /cart/{cart_uuid}/items/), not the request body. Passing it in the body is silently treated as a new cart creation — the caller receives a different cart_uuid with no error. This is not documented.

E-02 Step order is product-specific and not documented — must be inferred from required-fields Confirmed

GET /items/{item_uuid}/required-fields/ returns both which steps are needed and the order they must be called. Calling steps out of sequence returns a 422. This pattern works well but is entirely undocumented — we discovered it empirically after repeated failures.

E-03 GET /tickets returns ticket_types — PATCH /tickets expects tickets Confirmed

The response key and the request key use different names for the same concept. A consistent field name across both directions would make this step straightforward to implement.

E-04 Customer-info step returns 422 — expected payload not documented Sandbox?

PATCH /cart/{uuid}/customer-info/ with { first_name, last_name, email, phone } returns 422 Invalid request consistently. We could not determine the correct field names or whether ticket selection must be completed first. This may be a sandbox limitation or a documentation gap — either way it blocks step 9 entirely.

E-05 Create order returns "get() returned more than one Cart" Sandbox?

POST /attractions/booking/orders/ fails with an internal error indicating the cart lookup is returning multiple rows. Likely caused by multiple test carts created under the same sandbox consumer credentials — a sandbox data isolation issue rather than a real API bug.

Steps 1–8 confirmed working: cart creation, add item, required-fields, date selection, option selection, time slot, ticket quantities, language selection. The cart flow design is solid — the blockers are in the final steps and appear to be partly sandbox-related.
